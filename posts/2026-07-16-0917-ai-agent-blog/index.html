    더 이상 셀레니움으로 삽질하지 마세요: Python 'Browser-use'와 'CrewAI'로 구축하는 스스로 생각하는 웹 에이전트

    2026년 7월 16일

    지난 20년간 데이터 엔지니어링과 백엔드 아키텍처를 설계해 오면서, 제가 가장 기피했던 업무 중 하나는 바로 웹 스크래핑 및 RPA(로봇 프로세스 자동화) 파이프라인의 유지보수였습니다.

    Selenium이나 Playwright를 사용해 구축한 자동화 스크립트는 구축 당시에는 완벽해 보입니다. 하지만 서비스의 UI가 조금만 개편되거나, 마이너한 CSS 클래스명 하나만 바뀌어도 여지없이 NoSuchElementException을 뱉으며 멈춰버립니다. 이른바 **‘유지보수 지옥’**입니다. 웹사이트의 구조적 변화는 통제 불가능한 변수이기 때문에, 기존의 정적 룰 기반 RPA는 늘 깨지기 쉬운(fragile) 모래성 같았습니다.

    그러나 최근 인공지능 학계와 산업계의 화두인 **‘에이전트 기반 워크플로우(Agentic Workflow)’**는 이 문제를 근본적으로 해결하고 있습니다. 단순히 고정된 스크립트를 실행하는 것이 아니라, Vision 능력을 갖춘 LLM(대형 언어 모델)이 브라우저 화면을 직접 ‘보고’, ‘판단하여’, ‘행동하는’ 지능형 웹 에이전트의 시대가 열린 것입니다.

    오늘 소개해 드릴 Browser-use와 CrewAI의 조합은 이러한 패러다임 시프트를 상징하는 강력한 도구입니다. 이 글에서는 기존 RPA의 한계를 어떻게 극복하는지 그 메커니즘을 살펴보고, 현업에서 즉시 구동 가능한 엔터프라이즈급 아키텍처의 Python 코드 예제와 실무 적용 전략까지 깊이 있게 다뤄보겠습니다.

    1. 지능형 웹 에이전트의 메커니즘: Browser-use와 CrewAI의 시너지#

    전통적인 셀레니움 기반 자동화와 AI 에이전트 기반 자동화의 결정적인 차이는 **“컨텍스트 이해도”**에 있습니다.

    [전통적 RPA (Selenium/Playwright)]
DOM 변경 ──> Selector 매칭 실패 ──> 스크립트 중단 (에러 발생)

[지능형 웹 에이전트 (Browser-use + LLM)]
DOM 변경 ──> Screen Vision & DOM 트리 분석 ──> LLM의 재판단 ──> 새로운 경로로 자동 우회 실행

    Browser-use의 작동 원리#

    browser-use는 LLM이 웹 브라우저를 인간처럼 조작할 수 있도록 돕는 혁신적인 파이프라인입니다.

    1. 관찰(Observe): 현재 브라우저의 HTML DOM 트리 정보와 함께 실시간 스크린샷(Vision)을 LLM에 주입합니다.
    2. 생각(Reason): 사용자가 내린 최종 목표(예: “로그인 후 최신 리포트 다운로드”)와 현재 화면 상태를 비교하여 다음 행동을 계획합니다.
    3. 행동(Act): LLM은 단순 텍스트 답변 대신 "type": "click", "selector": "#submit-btn"과 같은 정형화된 액션 JSON을 반환하고, browser-use 내부의 Playwright 드라이버가 이를 수행합니다.

    CrewAI와의 유기적 결합#

    하지만 브라우저를 잘 다루는 것만으로는 복잡한 비즈니스 로직을 완벽히 수행할 수 없습니다. 데이터 수집 전략을 세우는 ‘기획자’, 브라우저를 조작해 데이터를 긁어오는 ‘수집가’, 그리고 수집된 날것의 데이터를 정제하여 슬랙이나 DB로 전송하는 ‘분석가’ 등 **역할 분담(Multi-Agent Orchestration)**이 필요합니다.

    CrewAI는 이러한 역할 분담을 오케스트레이션하는 프레임워크이며, Browser-use는 CrewAI의 에이전트들이 손에 쥐고 흔들 수 있는 **가장 강력한 도구(Tool)**가 됩니다.

    2. 실무 적용 가능한 지능형 에이전트 Python 구현 가이드#

    이제 개념을 넘어, 실제로 동작하는 파이프라인을 구축해 보겠습니다. 이번 예제에서는 **“Hacker News에 접속하여 AI 관련 트렌디한 뉴스 3개를 찾고, 각 기사의 핵심 내용을 요약하여 마크다운 리포트로 저장하는 타스크”**를 자동화해 보겠습니다.

    UI 구조가 바뀌더라도 LLM이 알아서 ‘AI’ 키워드를 찾고 브라우저를 조작하므로, 정적 셀렉터 선언이 전혀 필요 없습니다.

    사전 준비 및 환경 설정#

    우선 필요한 패키지를 설치합니다. 이 기술 스택은 Playwright 기반이므로 브라우저 바이너리 설치가 필수적입니다.

    pip install browser-use crewai langchain-openai python-dotenv playwright
playwright install

    프로젝트 루트 디렉토리에 .env 파일을 생성하고 OpenAI API 키를 입력합니다. (Vision 및 에이전트 추론 성능을 위해 gpt-4o 혹은 gpt-4o-mini 권장)

    OPENAI_API_KEY=your_openai_api_key_here

    구동 가능한 전체 Python 코드#

    import os
import asyncio
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from crewai import Agent, Task, Crew, Process
from browser_use import Agent as BrowserAgent
from crewai.tools import tool

# 1. 환경 변수 로드
load_dotenv()

# 2. LLM 초기화 (Vision 지원 모델 권장)
llm = ChatOpenAI(model="gpt-4o", temperature=0.0)

# 3. Browser-use를 CrewAI의 커스텀 Tool로 래핑
@tool("Web Browser Tool")
def browse_and_extract(instruction: str) -> str:
    """
    브라우저를 직접 실행하여 웹페이지를 탐색하고 정보를 추출하는 도구입니다.
    사용자의 자연어 지시사항(instruction)을 받아 직접 웹 서핑을 수행합니다.
    """
    # async 함수를 동기식 CrewAI Tool에서 실행하기 위한 이벤트 루프 처리
    async def run_browser():
        # browser-use 에이전트 생성
        browser_agent = BrowserAgent(
            task=instruction,
            llm=llm
        )
        result = await browser_agent.run()
        # 최종 결과 추출
        return result.final_result()

    try:
        loop = asyncio.get_event_loop()
    except RuntimeError:
        loop = asyncio.new_event_loop()
        asyncio.set_event_loop(loop)
        
    return loop.run_until_complete(run_browser())


# 4. CrewAI 에이전트 정의
# 역할 1: 웹을 탐색하여 원천 데이터를 수집하는 리서처
researcher = Agent(
    role="웹 트렌드 리서처",
    goal="Hacker News에서 최신 AI 관련 트렌드 뉴스를 탐색하고 원문 내용을 수집한다.",
    backstory="""당신은 깊이 있는 IT 트렌드 분석을 위해 웹 정보를 서치하는 전문 리서처입니다. 
    자연어로 브라우저 도구를 조작하여 정확한 타겟 데이터를 끈질기게 찾아냅니다.""",
    tools=[browse_and_extract],
    llm=llm,
    verbose=True
)

# 역할 2: 수집된 원문 데이터를 정제하고 요약본을 작성하는 라이터
writer = Agent(
    role="테크 에디터",
    goal="수집된 AI 뉴스 데이터를 기반으로 가독성 높은 요약 보고서를 마크다운 형식으로 작성한다.",
    backstory="""당신은 복잡한 기술 뉴스를 대중이 이해하기 쉽게 가공하는 20년 경력의 테크 에디터입니다.
    핵심 인사이트를 짚어내는 탁월한 요약 능력을 가지고 있습니다.""",
    llm=llm,
    verbose=True
)


# 5. 수행할 작업(Task) 정의
research_task = Task(
    description="""
    1. Hacker News(https://news.ycombinator.com)에 접속하세요.
    2. 'AI' 혹은 'LLM' 관련 헤드라인 기사 중 가장 흥미로운 3개를 선택해 클릭하세요.
    3. 각 기사의 제목과 본문 핵심 내용을 수집해 오세요.
    """,
    expected_output="3개 기사의 제목과 본문 텍스트 데이터 요약본",
    agent=researcher
)

write_task = Task(
    description="""
    리서처가 수집한 데이터를 바탕으로 다음 형식의 마크다운 보고서를 작성하고 'ai_trend_report.md' 파일로 저장하세요.
    
    # 주간 테크 브리핑: AI 트렌드 요약
    
    ## 1. [기사 제목 1]
    - **핵심 요약:** ...
    - **엔지니어 관점의 시사점:** ...
    
    ## 2. [기사 제목 2]
    ...
    """,
    expected_output="마크다운 형식의 보고서 작성 및 파일 저장 완료",
    agent=writer,
    output_file="ai_trend_report.md"
)


# 6. 크루(Crew) 구성 및 실행
def main():
    print("=== 지능형 웹 에이전트 루프 시작 ===")
    crew = Crew(
        agents=[researcher, writer],
        tasks=[research_task, write_task],
        process=Process.sequential, # 순차적 실행
        verbose=True
    )
    
    result = crew.kickoff()
    print("=== 작업 완료 ===")
    print(result)

if __name__ == "__main__":
    main()

    코드 분석 및 핵심 포인트#

    • @tool 데코레이터를 통한 연동: CrewAI 에이전트는 기본적으로 텍스트 기반 인터페이스입니다. browser-use 에이전트의 비동기 실행 루프를 browse_and_extract라는 동기식 커스텀 툴로 추상화하여 에이전트가 언제든지 브라우저를 제어할 수 있도록 무기를 쥐여주었습니다.
    • final_result() 반환: LLM 기반 브라우저가 화면 탐색 루프를 마치고 도출한 최종 텍스트 결과를 CrewAI 파이프라인의 후속 에이전트(writer)에게 그대로 전달합니다.

    3. 실무 아키텍처 관점에서의 예외 처리 및 엔터프라이즈 확장 전략#

    토이 프로젝트 수준의 코드와 상용 환경(Production)에서 동작하는 시스템의 차이는 **“예외 상황에 대한 견고한 대응”**에 있습니다. 시니어 관점에서 이 지능형 에이전트 시스템을 실제 서비스에 도입할 때 반드시 고려해야 할 세 가지 핵심 아키텍처 전략을 공유합니다.

    ① 비용(Token Cost) 및 속도 최적화#

    Browser-use는 매 스텝마다 DOM 트리와 스크린샷 이미지를 LLM에 전송합니다. 기가바이트 단위의 HTML을 그대로 보낸다면 토큰 비용 폭탄을 맞게 됩니다.

    • 해상도 및 DOM 필터링: browser-use는 기본적으로 불필요한 스크립트, CSS 스타일 태그를 제거하고 중요 텍스트 구조 위주로 DOM을 압축하여 전송합니다.
    • 하이브리드 LLM 전략: 복잡한 판단과 내비게이션에는 gpt-4o와 같은 고성능 Vision 모델을 사용하고, 수집된 텍스트의 요약 및 마크다운 변환 작업에는 비용이 100배 저렴한 gpt-4o-mini나 로컬 오픈소스 LLM(예: Llama-3)을 할당하는 Multi-LLM 라우팅 아키텍처를 구축해야 합니다.

    ② 사용자 세션 관리 및 로그인(Auth) 유지#

    매번 브라우저를 처음부터 실행하면 이메일 인증, OTP 등 자동 로그인 단계에서 병목이 생깁니다.

    • 브라우저 컨텍스트 보존: Playwright의 User Data Directory 옵션을 사용하여 브라우저 쿠키와 세션 정보를 로컬 파일 시스템에 유지합니다. 이렇게 하면 최초 1회만 수동 로그인을 해두면, 이후 에이전트는 로그인 상태가 유지된 상태로 작업을 시작할 수 있어 캡차(CAPTCHA) 등 보안 우회에 유리합니다.
    # browser-use 설정 예시 (컨텍스트 영속화)
from browser_use.browser.context import BrowserContextConfig

config = BrowserContextConfig(
    user_data_dir="/path/to/chrome/profile",  # 로컬 세션 저장소 지정
    highlight_elements=True                  # 실행 중 에이전트가 바라보는 영역 표시
)

    ③ 무한 루프 방지 및 가드레일(Guardrail) 설계#

    LLM은 가끔 엉뚱한 버튼을 계속 클릭하거나, 페이지 내에서 길을 잃고 헤매는 ‘Hallucination’을 보일 수 있습니다.

    • Max Steps 설정: 한 번의 태스크 수행 시 브라우저 액션의 최대 횟수(예: 15회)를 제한해야 합니다.
    • 자가 진입 검증(Self-Correction): 에이전트가 이전 액션과 동일한 액션을 3회 이상 반복할 경우, 현재 컨텍스트를 리셋하고 상위 프롬프트에 경고 피드백을 주어 탈출하게 유도하는 로직을 감시(Monitoring) 레이어에 심어야 합니다.

    결론: 단순 반복의 시대는 끝났습니다#

    우리가 오랜 시간 셀레니움의 CSS Selector 변경에 고통받으며 스크립트를 수정하던 시절은 저물고 있습니다.

    CrewAI와 Browser-use의 결합은 단순한 기술의 조합을 넘어, **“요구사항을 명시하면 도구를 사용하여 인간처럼 결과를 만들어내는 자동화 인프라”**의 시작점입니다. 이제 개발자는 깨지기 쉬운 규칙을 코딩하는 일에서 벗어나, 에이전트들의 페르소나를 설계하고, 이들이 원활히 협업할 수 있도록 조율하는 **‘오케스트레이터(Orchestrator)’**의 역할로 나아가야 합니다.

    지금 바로 여러분의 반복 업무 중 가장 유지보수가 번거로웠던 영역을 지능형 에이전트 아키텍처로 대체해 보십시오. 생산성의 도약은 생각보다 훨씬 가까이에 있습니다.
